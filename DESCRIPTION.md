# Описание приложения AI MCP Lab

Это учебное приложение на базе Laravel, которое демонстрирует принципы работы мультиагентных систем (MAS) и протокола управления инструментами (Model Context Protocol - MCP). Оно имитирует логику работы современного ИИ-агента: от получения запроса до поэтапного решения задачи с использованием инструментов и памяти.

## Основная концепция

Приложение работает по принципу **цепочки рассуждений (Chain of Thought)**. Вместо того чтобы дать мгновенный ответ, агент разбивает задачу на шаги:
1. **Thought (Мысль):** Планирование следующего действия.
2. **Call (Вызов):** Решение использовать конкретный инструмент (например, поиск).
3. **Observation (Наблюдение):** Получение результата от инструмента.
4. **Reflection (Рефлексия):** Анализ полученных данных.
5. **Answer (Ответ):** Формирование финального результата.

## Как это работает (Логика процесса)

1. **Запуск (API):** Пользователь отправляет запрос через `RunController@store`. Создается запись "Запуска" (`Run`).
2. **Инициализация Агента:** `AgentFactory` создает нужный тип агента (например, `ResearcherAgent`) и подключает к нему инструменты (`SearchTool`, `AgentTool`).
3. **Асинхронная очередь:** Чтобы не заставлять пользователя ждать, работа агента вынесена в фоновые задачи (`StepJob`).
4. **Цикл шагов:**
   - `StepJob` вызывает метод `processNextStep` у агента.
   - Агент анализирует последний сделанный шаг и решает, что делать дальше.
   - Каждый новый шаг сохраняется в базу данных в таблицу `steps`.
   - После создания шага агент снова ставит себя в очередь (`StepJob::dispatch`), пока не достигнет типа `answer` (финальный ответ) или лимита шагов.
5. **Память (Эмбеддинги):** При создании каждого шага `EmbeddingService` генерирует числовой вектор (эмбеддинг) контента. Это позволяет агенту "вспоминать" похожие ситуации из прошлого через поиск по вектору в базе данных (используется расширение `pgvector`).

## Ключевые классы и их роль

### Слой Агентов (`src/Services/Agents`)
*   **`AgentInterface`**: Стандарт, которому должен соответствовать любой агент. Определяет методы `run`, `processNextStep` и `retrieveFromMemory`.
*   **`NeuronAgent`**: Базовый класс с основной логикой. Он управляет циклом шагов, используя `LLMServiceInterface` для принятия решений, логированием в `AgentStep` и сохранением эмбеддингов.
*   **`ResearcherAgent`**: Специализированный агент-исследователь. Он переопределяет начало работы, чтобы сразу инициировать поиск информации.
*   **`AgentFactory`**: "Фабрика", которая решает, какого агента создать на основе типа запроса, и выдает ему набор инструментов.

### Инструменты (`src/Services/Tools`)
*   **`ToolInterface`**: Интерфейс для создания новых инструментов. Требует реализации `getName`, `getDescription` и `execute`.
*   **`SearchTool`**: Имитирует поиск в интернете.
*   **`AgentTool`**: Позволяет одному агенту делегировать задачу другому, создавая новый `Run`. Это основа мультиагентного взаимодействия.

### Инфраструктура и Интеллект
*   **`LLMServiceInterface`**: Интерфейс для взаимодействия с языковыми моделями.
*   **`MockLLMService`**: Продвинутая имитация ИИ. В отличие от простой логики, этот мок строит полный системный промпт и историю сообщений, имитируя процесс мышления реальной LLM.
*   **`EmbeddingServiceInterface`**: Интерфейс для генерации векторных представлений текста.
*   **`EmbeddingService`**: Фасад для работы с эмбеддингами, использующий `MockEmbeddingService` для генерации детерминированных векторов на основе текста (через CRC32).
*   **`StepJob`**: Фоновая задача Laravel, которая обеспечивает пошаговое выполнение логики агента без блокировки основного приложения.
*   **`RunController`**: Входная точка API для создания новых задач и просмотра логов выполнения.

## MCP Server (Model Context Protocol)

Приложение реализует стандарт **Model Context Protocol (MCP)** через Laravel AI SDK. Это позволяет выносить инструменты, ресурсы и промпты в отдельный слой, совместимый с современными ИИ-клиентами.

### Архитектура сервера (`src/app/Mcp`)

Центральным узлом является **`NeuronServer`** (`src/app/Mcp/Servers/NeuronServer.php`). Он регистрирует доступные возможности агента:

*   **Tools (Инструменты):** Исполняемый код, который агент может вызывать (например, `SearchTool`).
*   **Resources (Ресурсы):** Источники данных (файлы, записи БД), к которым агент может обращаться за контекстом (например, `MemoryResource`).
*   **Prompts (Промпты):** Шаблоны инструкций для модели (например, `SystemPrompt`).

### Как использовать и настраивать

1.  **Создание новых элементов:**
    Используйте встроенные команды Laravel для генерации:
    - `php artisan make:mcp-tool MyNewTool` — создать новый инструмент.
    - `php artisan make:mcp-resource MyNewResource` — создать новый ресурс.
    - `php artisan make:mcp-prompt MyNewPrompt` — создать новый шаблон промпта.

2.  **Регистрация в сервере:**
    Чтобы новый инструмент или ресурс стал доступен, его нужно добавить в соответствующие массивы (`$tools`, `$resources`, `$prompts`) внутри класса `NeuronServer.php`.

3.  **Логика работы:**
    - В **Tools** логика описывается в методе `handle(Request $request)`, а входные параметры определяются в `schema(JsonSchema $schema)`.
    - В **Resources** данные возвращаются через метод `handle(Request $request)`.
    - В **Prompts** текст инструкции задается в методе `handle(Request $request)`.

### Изменение существующей логики

- Чтобы изменить поведение поиска, отредактируйте `SearchTool.php`.
- Чтобы изменить логику извлечения памяти через векторный поиск, отредактируйте `MemoryResource.php`.
- Чтобы обновить системные инструкции для всех агентов, использующих MCP, измените `SystemPrompt.php`.

## Как подключить реальный ИИ

На данный момент приложение использует архитектуру, готовую к интеграции с реальными сервисами через интерфейсы.

### 1. Подключение LLM (например, OpenAI или Anthropic)
Вместо `MockLLMService` нужно создать реализацию `LLMServiceInterface`:
- Реализовать метод `generateNextStep`, который будет отправлять накопленную историю (`$run->steps`) в API (например, GPT-4).
- `MockLLMService` уже подготавливает системный промпт и историю сообщений, что упрощает переход.
- Использовать функцию **Function Calling**, чтобы модель сама возвращала JSON с именем инструмента и аргументами.

### 2. Подключение реальных эмбеддингов
Вместо `MockEmbeddingService` создайте свою реализацию `EmbeddingServiceInterface`:
- Замените генерацию псевдослучайных векторов на запрос к API (например, OpenAI `text-embedding-3-small` или локальную модель через Ollama).
- Это позволит поиску в `retrieveFromMemory` находить действительно похожие по смыслу шаги.

### 3. Настройка окружения
- Зарегистрируйте новые реализации в Service Provider.
- Добавьте API ключи в файл `.env`.
- Убедитесь, что база данных PostgreSQL имеет установленное расширение `pgvector` для хранения векторов.
